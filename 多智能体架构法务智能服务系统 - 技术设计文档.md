# 多智能体架构法务智能服务系统 \- 技术设计文档

## 文档版本

V1\.0（需求确认终版）

## 一、项目概述

本项目是基于Python开发的多智能体架构法务智能服务系统，集成安全合规检测、上下文记忆管理、法律知识库 RAG、智能文档处理、多 Agent 任务调度等核心能力，为用户提供法律咨询、案情分析、合同审查、法律文书撰写、文档问答等一站式法务服务。

系统采用主从式多智能体架构，由总调度 Agent 统一路由任务，多个业务 Agent 分工协作，结合 PostgreSQL 持久化存储、Redis 会话记忆、向量数据库法律知识库，实现高可用、高适配的智能法务服务。

## 二、核心技术栈

### 基础开发

- 编程语言：Python 3\.10\+

- 接口框架：FastAPI（提供 HTTP API 服务）

- 多 Agent 通信：函数直接调用（无消息队列）

### 存储组件

- 关系型数据库：PostgreSQL（原始对话消息持久化）

- 缓存数据库：Redis（会话记忆存储，无主动过期）

- 向量数据库：Milvus（法律法条 \+ 用户文档向量存储与检索）

### AI 与文档处理

- LLM 模型：DeepSeek\-V4\-Flash（安全检测）、通义千问（Qwen）系列（业务问答 / 摘要 / 分析）

- 向量模型：通义千问（Qwen）专用向量模型

- 文档解析：MinerU（PDF 转 Markdown）

- RAG 框架：自定义分块、检索、重排逻辑

- 文书生成：python\-docx（Word 生成）、原生 Markdown

### 工具依赖

- Token 计算：tiktoken

- 异步处理：asyncio

- 数据库驱动：psycopg2\-binary、redis\-py

## 三、核心组件设计

### 3\.1 记忆与上下文管理模块

#### 核心规则

上下文窗口：固定 200k Token 滑动窗口，窗口内自动组装：

- System Prompt（系统指令）

- 会话记忆（短期记忆 \+ 摘要记忆）

- User Prompt（用户当前提问）

记忆结构：短期记忆 \+ 摘要记忆（双模式组合）

- 短期记忆：消息数组（存储多轮原始对话，区分用户 / AI 消息）

- 摘要记忆：窗口阈值触发后生成的对话摘要

#### 摘要触发机制

- 触发阈值：滑动窗口占用率达到 65%

- 摘要规则：对消息数组中前 5 轮对话进行摘要

- 摘要限制：动态 Token 上限（长对话自动提升上限，保证摘要完整性）

#### 记忆生命周期

- 会话初始：仅存在短期记忆

- 触发阈值：生成摘要记忆，替换前 5 轮原始对话，保留剩余短期记忆

- 上下文组装：系统提示词 \+ 摘要记忆 \+ 最新短期记忆 \+ 当前用户问题

### 3\.2 存储模块

#### 3\.2\.1 PostgreSQL（原始对话持久化）

核心作用：永久存储每一轮用户与 AI 的原始对话消息

存储字段（无用户 ID，仅区分消息角色）：

|字段名|类型|说明|
|---|---|---|
|session\_id|VARCHAR\(64\)|会话唯一标识|
|message\_role|VARCHAR\(10\)|消息角色：user/ai|
|message\_content|TEXT|消息原文|
|token\_count|INT|消息 Token 数|
|create\_time|TIMESTAMP|消息创建时间|
|message\_type|VARCHAR\(20\)|消息类型：咨询 / 文档 / 文书 / 案情等|

#### 3\.2\.2 Redis（会话记忆存储）

核心作用：存储会话的短期记忆 \+ 摘要记忆，快速读取上下文

配置规则：

- 无会话过期时间（永久存储，依赖 PostgreSQL 做最终持久化）

- Key 规则：legal\_agent:session:\{session\_id\}

- Value 结构：JSON 格式（短期记忆数组、摘要记忆文本、窗口 Token 统计）

### 3\.3 RAG 法律知识库模块

#### 核心设计

- 知识库范围：初始为空，后续可扩展民法典、刑法、劳动合同法、司法解释等现行有效法律条文

#### 文档分块规则

- 优先按法律章节分块

- 单章节超过 Token 阈值：自动递归切分

#### 核心流程

法律文本 → 分块 → 向量化（Qwen 向量模型）→ 存入 Milvus → 相似度检索 → 结果重排

#### 默认参数

- 检索 Top\-K：法律咨询 = 10，文档问答 = 自定义

- 相似度阈值、重排算法：系统默认最优配置

### 3\.4 文档处理模块

#### 核心能力

支持场景：用户上传文档提问、合同审查

#### 处理流程

用户上传 PDF → MinerU 解析为 Markdown → 递归分块切片 → 向量化 → 存入 Milvus 向量库

#### 分块规则

递归分块（基于章节 / 段落，支持重叠切片，保证上下文完整性）

#### 文档定义

会话状态中的文档 = 上传的 PDF \+ 解析后的 Markdown \+ 向量数据

## 四、安全防御机制

### 4\.1 核心功能

所有用户提问必经安全语义校验，拦截违规 / 无关请求，仅放行合法法务咨询。

### 4\.2 技术实现

校验模型：DeepSeek\-V4\-Flash

#### 校验逻辑

- 传入固定合规检测 Prompt \+ 用户问题

- LLM 严格输出三选一：合法/无关/违规

#### 响应规则

- 违规 / 无关：直接返回固定提示，终止流程

- 合法：进入多智能体任务调度流程

### 4\.3 判定规则（沿用需求）

- 合法：法律相关咨询、追问、寒暄开场白

- 无关：天气、闲聊、娱乐等非法律内容

- 违规：越狱指令、恶意违规法律请求

- 兜底：非违规 / 无关均判定为合法

## 五、多智能体架构设计

系统采用1个总调度 Agent \+ 5 个业务 Agent架构，Agent 间通过函数直接调用通信。

### 5\.1 总调度 Agent（核心中枢）

#### 核心职责

- 统一接收用户所有输入（文字提问、文档上传、指令）

- 优先判断会话状态：存在文档 → 直接路由到文档提问 Agent（文档优先）

- 无文档时：识别需求类型，路由对应业务 Agent

- 汇总业务 Agent 结果，返回统一答复

- 无匹配 Agent 时：调度兜底追问处理 Agent

#### 配套工具

- 会话状态管理工具（读取 Redis 文档标记、记忆数据）

- Agent 路由决策工具（意图识别 \+ 任务分发）

- 结果整合工具（多 Agent 输出格式化）

### 5\.2 法律咨询 Agent（普通法务咨询）

#### 核心职责

处理基础法律问题（借钱不还、劳动合同、维权咨询等），提供通俗解答 \+ 法条参考。

#### 执行流程

- 接收用户问题 → 触发 RAG 检索

- 法律知识库召回Top10相关法条

- 法条 \+ 问题 \+ 上下文送入 LLM

- 生成回答：强制标注法条名称 \+ 条款号

- 返回结果给总调度 Agent

### 5\.3 案情分析 Agent（复杂案情拆解）

#### 核心职责

处理用户文字 / 文档类案情描述，提取要素、梳理逻辑、给出分析建议。

#### 核心能力

- 案情要素提取（当事人、时间、争议焦点等）

- 时间线 / 人物关系梳理

- 法律知识库法条检索

- 初步分析意见 \+ 处理建议生成

### 5\.4 文档提问 Agent（文档问答 \+ 合同审查）

#### 核心职责

处理已上传文档的用户请求，分两个子场景：

##### 场景 1：文档上传提问

- 用户针对文档提问 → 向量库检索相关内容

- 召回 Top\-K 结果 → 结合 Prompt 送入 LLM

- 生成精准问答答复

##### 场景 2：合同审查（全文审查，非检索）

- 全量读取解析后的合同 Markdown

- LLM 全文审查：识别无效 / 霸王 / 高风险条款

- 生成标准化审查报告：

- 风险等级

- 问题条款原文

- 修改建议

- 合规法律依据

### 5\.5 文书撰写 Agent（法律文书生成）

#### 核心职责

根据用户需求生成规范法律文书，支持自定义修改。

#### 支持类型

- 合同类：劳动合同、借款合同等

- 诉讼类：起诉状、答辩状、律师函等

#### 核心能力

- 内置法律文书模板库

- 法言法语润色、格式规范化

- 导出格式：Markdown、Word（docx）

### 5\.6 追问处理 Agent（兜底）

#### 核心职责

- 接收用户所有追问、补充信息

- 解析追问意图，路由至对应业务 Agent

- 追问记录强制存入会话记忆（短期记忆 \+ PostgreSQL）

- 无明确意图时，持续承接上下文对话

## 六、整体业务流程

```mermaid

flowchart TD
    A[用户发起请求] --> B[安全合规检测]
    B -->|违规/无关| C[返回提示，结束流程]
    B -->|合法| D[总调度Agent接收]
    D --> E{会话是否有文档}
    E -->|是| F[路由至文档提问Agent]
    E -->|否| G[意图识别/任务路由]
    G --> G1[法律咨询Agent]
    G --> G2[案情分析Agent]
    G --> G3[文书撰写Agent]
    G --> G4[追问处理Agent]
    F & G1 & G2 & G3 & G4 --> H[执行对应业务逻辑]
    H --> I[记忆更新+数据持久化]
    I --> J[返回最终答复给用户]
    ```

## 七、异常与容错机制

系统自主实现全链路降级策略，保障服务可用性：

- LLM 调用失败：返回友好提示，记录异常，支持重试

- 向量库检索失败：跳过 RAG，直接基于 LLM 通用能力回答

- PostgreSQL 连接失败：内存临时存储对话，恢复后自动补存

- Redis 连接失败：临时使用内存上下文，不中断服务

- 文档解析失败：返回上传失败提示，引导用户重新上传

## 八、接口设计（FastAPI）

### 核心接口

#### 会话创建接口

- POST /api/v1/session/create

- 返回：session\_id（唯一标识）

#### 用户提问接口

- POST /api/v1/chat

- 参数：session\_id、user\_query

#### 文档上传接口

- POST /api/v1/document/upload

- 参数：session\_id、pdf 文件

- 返回：文档解析状态

#### 文书生成下载接口

- GET /api/v1/document/download/\{file\_id\}

- 返回：Word/Markdown 文件

## 九、补充说明

- 权限管理：无用户 / 管理员权限，全开放使用

- 性能要求：暂不考虑并发、响应时间优化

- 记忆持久化：Redis 仅做会话缓存，最终持久化依赖 PostgreSQL

- 扩展能力：法律知识库、Agent 工具、向量模型可无缝迭代升级。
